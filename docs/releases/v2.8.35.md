# v2.8.35 - Deterministic backend audio start sequence

This UI-only release removes the experimental browser-side live.wav 409 reset/retry logic from v2.8.32/v2.8.33 and replaces it with the call order proven by the backend audit.

## Behavior

The pass-row backend test audio path now uses this sequence:

1. Stop any previous browser audio session.
2. POST `/api/radio/audio/live/stop` as best-effort cleanup.
3. Plan tuning/Doppler as best effort.
4. POST `/api/radio/audio/live/start?...` and require success.
5. Open `/api/radio/audio/live.wav?...` only after start succeeds.
6. Stop audio only when the operator presses Stop/closes the modal or a real stream error occurs.

If live.wav returns HTTP 409, the response body is shown in the Backend Test modal/status area. The browser does not hide that state by repeatedly stopping/restarting behind the scenes.
