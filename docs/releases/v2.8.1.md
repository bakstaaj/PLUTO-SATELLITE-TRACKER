# Pluto Satellite Tracker v2.8.1 Runtime Startup Guard

This maintenance update makes the recovered v2.8 runtime layout repeatable at startup.

The Pluto backend continues to use persistent application files under `/mnt/jffs2/pluto_sat_tracker` and transactional runtime files under `/tmp/pluto_sat_tracker`.

The startup guard recreates volatile runtime prerequisites before the backend starts:

- `/tmp/pluto_sat_tracker/data`
- `/tmp/pluto_sat_tracker/logs`
- `/tmp/pluto_sat_tracker/tmp`
- `/tmp/pluto_sat_tracker/tools -> /mnt/jffs2/pluto_sat_tracker/tools`
- optional Python runtime links when persistent runtime directories exist
- a temporary `satellites.json` seed when a persistent catalog copy is available
- the app-root backend wrapper pointing to `/mnt/jffs2/pluto_sat_tracker/bin/pluto_sat_tracker`

Validation is performed by `tools/repair_pluto_tmp_runtime_v2_8_1.sh` and requires:

- Pluto SSH access
- `/api/status` reporting `ok=true`
- backend version not reporting `0.1.0`
- `/api/status` reporting `/tmp/pluto_sat_tracker/data`
- `/api/refresh/passes` returning HTTP 200
- refresh worker reaching `state=ok`
- `/api/passes` returning a non-trivial payload
