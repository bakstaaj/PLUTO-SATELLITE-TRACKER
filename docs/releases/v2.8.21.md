# v2.8.21 Protocol Resolver

Adds a browser-side protocol resolver that keeps modulation, protocol hints, and decoder profile separate.

## Why

The pass list had correct visible mode/downlink information, but selected `pass.radio` could point at a different transmitter on multi-transmitter satellites. That caused labels and future decoder routing to collapse into broad families such as APRS, Packet, Digital, or CW.

## Behavior

- Preserves literal modulation from the selected pass/transmitter: CW, AFSK, FSK, GMSK USP, BPSK, QPSK, DVB-S2, DUV, USB, etc.
- Keeps APRS/AX.25/USP/telemetry/voice as protocol hints, not as replacements for modulation.
- Adds a normalized `protocol_resolver` object to each pass in the browser:
  - `modulation`
  - `protocol_hint`
  - `decoder_profile`
  - `button_label`
  - `downlink_hz`
  - selected transmitter notes
- Uses the resolved downlink/mode for pass-list action labels, radio target display, audio start, and Doppler tracking payload.
- Does not change backend C or claim support for decoders that are still pending.

## Examples

- AFSK + APRS hint => label `Receive: AFSK`, decoder profile `afsk1200_ax25_aprs`.
- FSK + AX.25 hint => label `Receive: FSK`, decoder profile `fsk_symbol_clock_slicer_pending`.
- GMSK USP => label `Receive: GMSK USP`, decoder profile `gmsk_symbol_clock_slicer_pending`.
- USB => label `Listen`, decoder profile `analog_listen`.
