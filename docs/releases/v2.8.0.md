# Pluto Satellite Tracker v2.8.0

## Summary

v2.8.0 stabilizes the recovered Pluto web UI and codifies the runtime repair that moves generated/transactional files out of SD-card storage and into `/tmp/pluto_sat_tracker`.

## What changed

- Restored the asset-split web UI deployment so the browser loads `web/index.html`, `web/app.js`, and `web/app.css` consistently.
- Preserved the UI version badge and current pass/listen/decode frontend behavior from the deployed working Pluto copy.
- Standardized runtime generated data under `/tmp/pluto_sat_tracker/data`.
- Standardized runtime logs under `/tmp/pluto_sat_tracker/logs`.
- Added a reusable repair utility: `tools/repair_pluto_tmp_runtime_v2_8.sh`.
- The repair utility recreates the `/tmp` runtime layout, links persistent tools/runtime folders, seeds `satellites.json`/`passes.json` when available, rewrites the Pluto startup wrapper, restarts the backend, and validates `/api/status` plus `/api/refresh/passes`.

## Root cause fixed

The app became split across several runtime roots:

- persistent app files in `/mnt/jffs2/pluto_sat_tracker`
- older generated files under `/media/mmcblk0p1/pluto_sat_tracker/data`
- desired transactional files under `/tmp/pluto_sat_tracker/data`

When `--data-dir` was moved to `/tmp/pluto_sat_tracker/data`, the backend derived its refresh runner root from the parent of `--data-dir`, so it expected:

```text
/tmp/pluto_sat_tracker/tools/pluto_refresh_data.sh
/tmp/pluto_sat_tracker/python
/tmp/pluto_sat_tracker/python-runtime
/tmp/pluto_sat_tracker/data/satellites.json
```

The repair utility recreates that expected runtime shape while keeping persistent files on flash/SD and generated files in `/tmp`.

## Validation

Validated on Pluto LAN address `192.168.68.104`:

- `/api/status` reports a non-0.1.0 backend.
- `/api/status` reports `data_dir=/tmp/pluto_sat_tracker/data`.
- `/api/refresh/passes` returns HTTP 200.
- Pass generation works after restoring/rebuilding the satellite catalog.
- Browser UI loads the asset-split frontend and displays the improved UI.

## Operational note

If Pluto is power-cycled, `/tmp` is rebuilt by `run_tracker.sh`. Persistent catalog/pass seeds remain outside `/tmp`; the repair utility can be rerun from the host if a runtime mismatch is observed again.
